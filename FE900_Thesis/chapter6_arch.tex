\chapter{High-Level Architecture of the FPGA-Based Matching Engine}
\label{ch:architecture}

The design of an ultra-high-frequency trading (ultra-HFT) system is predicated on the dual mandates of \textbf{nanosecond-level determinism} and \textbf{minimal jitter}. Unlike software-based architectures that rely on general-purpose CPUs with non-deterministic branch prediction and cache hierarchies, an FPGA-based matching engine is architected as a hardware-level pipeline. This chapter details the high-level architecture of the proposed system, decomposing it into a series of specialized, heterogeneous logic modules implemented in Register-Transfer Level (RTL). \section{Modular Design and Functional Decomposition}

The system architecture employs a \textbf{deeply pipelined design pattern}, where each stage performs a discrete transformation on the data stream within a fixed number of clock cycles. This allows the system to achieve a high operating frequency ($f_{clk} \geq 322.26$ MHz for 10GbE) and ensures that the \textbf{Worst-Case Execution Time (WCET)} is identical to the average-case execution time.

\subsection{Ingress and Protocol Parsing}
The ingress stage is engineered to consume raw Ethernet frames at line-rate (10 Gbps) with zero packet loss. \datamethodology
\begin{description}
    \item[RX Parser (\texttt{rx\_parser}):] This module is implemented as a high-performance \textbf{Finite State Machine (FSM)}. It inspects the 64-bit data bus from the MAC interface. The FSM transitions through states to peel away the encapsulation layers:
    \begin{itemize}
        \item \textbf{ETH\_HEAD:} Extracts the 14-byte Ethernet header and validates the EtherType (0x0800 for IPv4).
        \item \textbf{IP\_HEAD:} Parses the 20-byte IPv4 header, extracting the Source IP and verifying the Protocol field (17 for UDP).
        \item \textbf{UDP\_HEAD:} Identifies the 8-byte UDP header, specifically checking the \textbf{Destination Port} to filter for relevant multicast feeds.
    \end{itemize}
    The parser utilizes \textbf{AXI-Stream (AXI-S)} signals for flow control:
    \begin{itemize}
        \item \texttt{TVALID}: Asserted when the parser has valid data for the decoder.
        \item \texttt{TREADY}: Asserted by the decoder when it can accept data, implementing backpressure.
        \item \texttt{TLAST}: Asserted at the final word of an Ethernet frame to signal the end of the packet boundary.
    \end{itemize}

    \item[ITCH Decoder (\texttt{itch\_decoder}):] This module decodes the \textbf{NASDAQ ITCH 5.0} binary protocol. It employs a \textbf{barrel shifter} and a sliding-window buffer to handle unaligned message boundaries across AXI-Stream words. The decoder extracts specific fields from "Add Order" (Type 'A') and "Order Executed" (Type 'E') messages:
    \begin{equation}
        \text{Tick}_{norm} = \{ \text{Price}, \text{Quantity}, \text{Side}, \text{OrderID} \}
    \end{equation}
    The output is a normalized 128-bit internal bus structure, abstracting the wire protocol from the downstream strategy logic.
\end{description}

\subsection{State Management and Decision Logic}
The core intelligence of the system resides in the synchronization of market state and reinforcement learning inference.

\begin{description}
    \item[Hardware Order Book (\texttt{book2}):] The order book maintains the \textbf{Best Bid and Offer (BBO)} state. To achieve single-cycle updates, it utilizes a \textbf{dual-port BRAM (Block RAM)} or Distributed RAM for price level storage. When a Tick arrives, the module performs an instantaneous lookup and update:
    \begin{equation}
        P_{best} = \begin{cases} \max(P_{bid}) & \text{for Buy side} \\ \min(P_{ask}) & \text{for Sell side} \end{cases}
    \end{equation}
    The module also calculates the \textbf{Order Book Imbalance (OBI)}, as defined in Chapter \ref{ch:microstructure}, and streams the updated BBO state to the RL Core.

    \item[RL Strategy Core (\texttt{strat\_decide}):] This is the primary decision engine, implementing the synthesized DRL policy. It is architected as a \textbf{systolic array} of Multiply-Accumulate (MAC) units. The inference logic is fully unrolled and pipelined, ensuring that once the first layer of the neural network receives the BBO features, the final quoting decision is emitted exactly $N$ cycles later, where $N$ is the depth of the network. This provides the "Godhood" level of latency needed to front-run software competitors.
\end{description}

\subsection{Egress and Execution}
Upon a trade signal, the egress stage generates and transmits the response packet.

\begin{description}
    \item[OUCH Encoder (\texttt{order\_encode}):] Translates internal trade signals into \textbf{NASDAQ OUCH 5.0} messages. It performs field-packing and \textbf{Endianness conversion} (Big-to-Little) in hardware. The encoder generates the "Enter Order" message, populating the \texttt{UserRef}, \texttt{Buy/Sell Indicator}, \texttt{Quantity}, and \texttt{Price} fields.

    \item[TX Bridge (\texttt{tx\_bridge}):] The final stage adapts the encoder output to the 10GbE MAC. It inserts the required \textbf{Inter-Frame Gap (IFG)} and manages the Ethernet preamble. Crucially, it ensures that the physical layer interface (XGMII) is never starved of data during a transmission burst, maintaining a deterministic \textbf{Wire-to-Wire} latency.
\end{description}

\section{Latency Decomposition and Determinism}
The total \textbf{Tick-to-Trade (T2T)} latency, $\mathcal{L}_{total}$, is the sum of the latencies of each constituent module. Because the design is synchronous and non-blocking, the total latency is theoretically constant:
\begin{equation}
    \mathcal{L}_{total} = \sum_{i \in \text{modules}} \frac{C_i}{f_{clk}} + \mathcal{L}_{PHY}
\end{equation}
where:
\begin{itemize}
    \item $C_i$: The fixed number of \textbf{clock cycles} required by module $i$ to process a message.
    \item $f_{clk}$: The system clock frequency (322.26 MHz).
    \item $\mathcal{L}_{PHY}$: The fixed latency of the Physical Medium Attachment (PMA) and Physical Coding Sublayer (PCS), typically $\approx 100$ ns.
\end{itemize}
In this architecture, $C_{parser}=4$, $C_{decoder}=6$, $C_{book}=2$, $C_{RL}=12$, and $C_{encoder}=5$, resulting in an internal logic latency of $\approx 90$ ns. Including the PHY, the end-to-end T2T latency is verified at \textbf{$< 200$ ns}.

\subsection{Empirical Analysis via Hardware Timestamping}
To verify this theoretical determinism and analyze the real-world latency distribution, we utilize high-precision hardware timestamps captured at the ingress and egress points of the FPGA fabric. By recording the nanosecond-precision time of arrival ($T_{ingress}$) and departure ($T_{egress}$) for each transaction, we derive the empirical latency:
\begin{equation}
    \Delta T = T_{egress} - T_{ingress}
\end{equation}
This timestamp-based approach allows us to construct a \textbf{Probability Density Function (PDF)} of the latency. Unlike software-based systems that exhibit significant "jitter" (variance) due to OS interrupts and cache misses, our timestamp analysis confirms that the latency distribution is tightly clustered around the mean, with a standard deviation approaching zero. This empirical validation proves that the system's performance remains invariant even under peak market load (see Chapter \ref{ch:results} for detailed benchmarks).

\section{Software Support and Hybrid Co-Design}
While the critical path is in hardware, the system relies on a high-performance software stack for management and observability.
\begin{itemize}
    \item \textbf{Kernel Bypass via VFIO-PCI:} The software layer communicates with the FPGA via PCIe Gen4 x16. Using the \texttt{vfio-pci} driver, the application maps the FPGA's BAR (Base Address Register) into user-space, allowing for \textbf{Zero-Copy} telemetry collection and model weight updates.
    \item \textbf{Cycle-Accurate Simulator:} Prior to hardware synthesis, the RTL is validated against a bit-accurate C++ simulator. This ensures that the reinforcement learning policy's behavior in the hardware systolic array perfectly matches its performance in the training environment.
\end{itemize}

\section{Conclusion}
The FPGA architecture detailed in this chapter provides the deterministic substrate required for ultra-HFT. By offloading the entire execution pipeline, spanning from parsing to RL-based decision making, into hardware logic, we eliminate the bottlenecks of traditional software trading engines.
 This architecture forms the basis for the performance benchmarks and comparative analysis presented in the final chapters of this thesis.
